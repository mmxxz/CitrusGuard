# Agent 重构计划 (TODO)

本文档旨在跟踪和规划 Agent 系统的重构过程。该计划基于一个核心思想：**从一个简单的、由 LLM 驱动的流程，升级为一个由“智能路由”和“多维证据置信度”驱动的、更像专家的诊断框架。**

## 1. 现状分析与核心维度评估

### 1.1. 总体概览

当前的 Agent 系统基于 `LangGraph` 构建，其核心是一个由初始意图识别驱动的条件路由图。

-   **工作流**: `意图识别 (diagnosis/treatment/general_qa)` -> `分支执行`。
-   **诊断分支**: 线性地执行 `信息收集` -> `LLM评估置信度` -> `(可选) LLM生成追问` -> `生成报告`。
-   **主要缺陷**:
    1.  **置信度黑盒**: 置信度由 LLM 主观判断，不可靠且无法解释。
    2.  **追问盲目**: 追问并非为了填补特定的证据缺口，效率低下。
    3.  **路由简单**: 无法区分复杂诊断和简单的知识问答，所有问题都可能走入复杂流程，造成资源浪费。
    4.  **RAG 融合浅**: 知识库的利用停留在简单的上下文填充，未能实现结构化的证据匹配。

### 1.2. 核心系统维度评估

为了更精确地指导重构，我们从六个核心维度对现有系统进行深入评估：

| 维度 | 安排妥当性 | 闭环情况 | 核心评价与优化方向 |
| :--- | :--- | :--- | :--- |
| **Tool (工具)** | ✅ 高 | 🌗 **部分闭环** | 结构良好，但依赖模拟数据。**优化方向**: 替换模拟工具，与真实世界数据打通 (**任务 3.1**)。 |
| **Agent (代理)** | ✅ 高 | ❌ **逻辑未闭环** | 技术架构优秀，但核心决策逻辑是黑盒。**优化方向**: 引入可解释的决策逻辑 (**任务 3.2, 3.3**)。 |
| **Memory (记忆)** | ✅ 高 | ✅ **会话内闭环** | 短期记忆机制稳健可靠，**无需重构**。 |
| **Session (会话)** | ✅ 高 | ✅ **完全闭环** | 会话管理是系统最稳固的部分，**无需重构**。 |
| **Plan (计划)** | ❌ 低 | ❌ **完全开环** | 计划能力非常初级，无动态调整能力。**优化方向**: 本次重构暂不涉及，未来可增强。 |
| **RAG (检索)** | ✅ 高 | 🌗 **机制闭环，认知开环** | 数据流完整，但知识未被深度利用。**优化方向**: 建立结构化知识库并进行证据匹配 (**任务 1.2, 3.2**)。 |

### 1.3. 多轮对话处理评估

-   **现状**: 系统通过 API (`/continue`) 和 LangGraph 的检查点 (`MemorySaver`) 机制来处理多轮对话。Agent 提问后会暂停，用户的回答会唤醒 Agent 并让其继续执行。
-   **评估**:
    -   **交互闭环 (是)**: 从技术和交互流程上看，`Agent提问 -> 用户回答 -> Agent接收` 的机制是完整且可靠的。
    -   **认知闭环 (否)**: 这是当前的核心短板。Agent 将对话历史作为非结构化的文本日志进行处理，每次都需重读全部上下文。它无法针对用户的回答进行**结构化的认知更新**（例如，将回答“叶片是黄色的”转化为对“证据矩阵”中一个具体字段的更新）。
-   **优化方向**: 引入 `EvidenceMatrix` 作为核心状态。用户的回答将被用来**精确填补证据矩阵中的空白**，而不是简单地追加到聊天记录中。这使得 Agent 的每一步推理都基于一个动态更新的、结构化的事实基础，从而**闭合认知循环** (**任务 3.1, 3.2**)。

## 2. 重构计划与任务排期

我们将按照“**先基础，后路径，先简单，后复杂**”的原则，分阶段进行重构。

**目标**: 搭建新框架所需的核心数据结构，并升级系统的“大脑中枢”——智能路由网关。

    -   [ ] 在 `app/schemas/` 中创建 `disease_profile.py`，定义 `DiseaseProfile` (病害特征档案) 的结构，包含 `visual_symptoms_checklist`, `environmental_triggers_checklist` 等。
    -   [ ] 在 `app/core/orchard_state.py` 中更新 `OrchardState`，使其能够容纳 `EvidenceMatrix`。

-   [ ] **任务 1.2: 创建病害特征档案的存储 (解决 RAG 认知开环)**
    -   [ ] 在 `app/models/` 中创建 `disease_profile.py`，定义对应的 SQLAlchemy 模型。
    -   [ ] 在 `app/crud/` 中创建 `disease_profile.py`，提供对病害档案的增删改查功能。
    -   [ ] 创建一个新的 Alembic migration 脚本，用于在数据库中生成 `disease_profiles` 表。
    -   [ ] **(数据填充)** 编写一个一次性脚本，用于将现有的非结构化知识，转化为结构化的 `DiseaseProfile` 并存入数据库。

-   [ ] **任务 1.3: 升级智能路由网关**
    -   [ ] 修改 `app/agents/graph/intent_recognition_node.py`。
    -   [ ] 更新 LLM 提示，使其能够按新标准分类意图: `Complex Diagnostic`, `Simple Q&A`, `Direct Tool-Use`。

### **阶段二：实现路径 B - 快速问答与工具调用 (快速通道)**

**目标**: 构建高效处理简单请求的路径，快速交付价值，并验证新路由的有效性。

-   [ ] **任务 2.1: 实现 RAG 快速问答节点**

    -   [ ] 该节点实现精简的 RAG 流程：`接收问题` -> `向量检索` -> `LLM 结合 Top-K 结果生成综合答案` -> `存入 State`。
    -   [ ] 在 `graph.py` 中，将路由网关的 `Simple Q&A` 意图连接到此节点。

-   [ ] **任务 2.2: 实现直接工具调用节点**

    -   [ ] 在 `graph.py` 中，将路由网关的 `Direct Tool-Use` 意图连接到此节点。

### **阶段三：实现路径 A - 复杂诊断流程重构 (核心升级)**

**目标**: 彻底改造诊断流程，用基于证据的、可解释的置信度模型取代旧的黑盒模型。

-   [ ] **任务 3.1: 重构证据收集工具 (解决 Tool 部分闭环 & 认知闭环)**
    -   [ ] **(视觉证据)** 升级 `run_image_diagnosis.py`。将其从模拟节点改造为调用真实的多模态模型，并要求模型输出结构化的 JSON (符合 `EvidenceMatrix` 中的视觉证据部分)。
    -   [ ] **(症状证据)** 创建新工具 `extract_symptom_evidence.py`，使用 LLM 从用户文本中提取结构化的症状描述。
    -   [ ] **(环境/历史证据)** 修改 `fetch_weather_data.py` 和 `retrieve_historical_cases.py`，确保它们的输出能被正确格式化并填入 `EvidenceMatrix`。

-   [ ] **任务 3.2: 实现置信度计算节点 (解决 Agent 逻辑未闭环)**
    -   [ ] 创建一个全新的、**非 LLM 的**逻辑节点 `calculate_confidence_node.py`。
    -   [ ] **核心逻辑**:
        1.  从 `OrchardState` 中读取 `EvidenceMatrix`。
        2.  从数据库中获取 Top-K 候选病害的 `DiseaseProfile`。
        3.  循环对比证据和档案，根据预设权重计算量化置信度分数。
        4.  识别出“证据缺口”（档案中有但证据中无的关键特征）和“差异化诊断点”（多个高分候选病害的核心区别）。
        5.  将结构化的置信度结果、证据缺口等存入 `OrchardState`。

-   [ ] **任务 3.3: 实现基于“证据缺口”的智能追问 (解决 Agent 逻辑未闭环)**
    -   [ ] 重构 `initiate_clarification.py` 节点。
    -   [ ] 新的提示将以 `calculate_confidence_node` 输出的“证据缺口”和“差异化诊断点”作为**强约束**输入。
    -   [ ] LLM 的任务不再是自由发挥，而是**根据这些具体的技术点生成精准的、用于排除或确认的澄清问题**。

-   [ ] **任务 3.4: 重新组装诊断图**
    -   [ ] 在 `graph.py` 中，彻底重构旧的诊断分支。
    -   [ ] 新流程: `路由网关` -> `并行证据收集` -> `置信度计算` -> `条件路由 (根据置信度阈值)` -> `智能追问 (如果置信度低)` 或 `生成报告 (如果置信度高)`。

### **阶段四：收尾与测试**

## 4. 存留问题与解决方案

*(记录在开发过程中遇到的挑战和解决思路)*

-   **问题**: 数据库迁移时遇到 LangChain 相关表的依赖问题
-   **解决方案**: 修改了 Alembic 迁移脚本，移除了对 LangChain 表的删除操作，避免依赖冲突

-   **问题**: RAG 节点中的向量存储导入问题
-   **解决方案**: 修复了 `rag_qa_node.py` 中的导入，使用 `get_vector_store()` 函数而不是直接导入 `vector_store`

-   **问题**: 测试时需要 thread_id 配置
-   **解决方案**: 在测试脚本中添加了 `configurable` 配置，提供 `thread_id` 参数

-   **问题**: 部分工具调用需要真实的果园数据
-   **解决方案**: 添加了错误处理机制，当缺少必要数据时提供友好的错误信息

-   **问题**: LangChain 版本兼容性警告
-   **解决方案**: 这些是弃用警告，不影响功能，建议后续升级到新版本的 LangChain 包

-   **问题**: Pydantic V2 弃用警告
-   **解决方案**: 将 `.dict()` 方法替换为 `.model_dump()` 方法，符合 Pydantic V2 规范

-   **问题**: 结构化数据解析复杂性
-   **解决方案**: 创建了专门的数据提取函数，分别处理视觉症状、环境因素、症状发展模式和治疗方案，确保数据完整性和准确性

-   **问题**: 数据量较大（52个条目）
-   **解决方案**: 实现了批量处理机制，添加了进度显示和错误处理，确保所有数据都能正确处理

-   **问题**: LangGraph 检查点序列化递归限制错误
-   **解决方案**: 实现了自定义序列化器 `CustomSerializer`，能够清理 SQLAlchemy 对象和循环引用，确保状态对象能够正确序列化到检查点中

-   **问题**: 天气节点需要接入真实天气API
-   **解决方案**: 集成了 OpenWeatherMap API，支持基于果园坐标和地址的天气查询，提供实时天气数据和5天预报，包含温度、湿度、风速、降雨概率等详细信息

-   **问题**: 前端显示"Could not generate briefing"错误
-   **解决方案**: 为所有工具响应（天气、RAG QA、历史案例、果园档案）添加了 `briefing` 字段，确保前端能够正确显示工具执行结果

-   **问题**: 天气工具只能获取到果园ID，无法获取完整果园信息进行位置查询
-   **解决方案**: 修改天气工具逻辑，在查询天气前先调用 `fetch_orchard_profile` 获取完整果园信息，并改进了城市名称提取算法，支持更多地址格式

-   **问题**: 天气信息在前端显示时没有换行，所有信息挤在一行
-   **解决方案**: 将天气摘要中的换行符从 `\n` 改为HTML的 `<br/>` 标签，确保前端能够正确显示多行格式

## 5. 重构总结

本次 Agent 重构成功实现了从简单的 LLM 驱动流程升级为基于"智能路由"和"多维证据置信度"的专家诊断框架。主要成果包括：

### 5.1. 核心架构升级
- **智能路由网关**: 支持三种意图分类（Complex Diagnostic、Simple Q&A、Direct Tool-Use）
- **证据矩阵系统**: 结构化的四维证据收集（视觉、症状、环境、历史）
- **置信度计算引擎**: 非LLM的量化匹配算法，提供可解释的推理过程
- **智能追问系统**: 基于证据缺口的精准问题生成

### 5.2. 技术实现
- **数据模型**: 完整的 Pydantic 数据结构和数据库模型
- **路由逻辑**: 基于置信度的条件路由决策
- **工具集成**: 统一的工具调用接口和错误处理
- **测试覆盖**: 端到端测试验证所有核心功能
- **数据集成**: 成功集成了52个病虫害的结构化数据，包括症状、病原、发生规律和防治方法
- **天气API集成**: 集成了 OpenWeatherMap API，支持基于果园位置的实时天气查询，提供当前天气和5天预报数据

### 5.3. 系统优势
- **可解释性**: 所有决策过程都有明确的逻辑依据
- **效率提升**: 简单问题走快速通道，复杂问题走深度分析
- **扩展性**: 模块化设计便于后续功能扩展
- **可靠性**: 非LLM逻辑减少了不确定性，提高了诊断准确性

### 5.4. 后续优化方向
- 集成真实的图像诊断API
- 完善环境数据获取工具
- 优化置信度计算算法
- 增加更多病害档案数据（当前已有52个，可继续扩展）
- 实现用户反馈学习机制
- 优化数据提取算法，提高结构化数据的解析准确性
- 添加更多环境触发因素和症状发展模式数据
